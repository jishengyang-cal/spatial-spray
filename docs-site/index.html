Cross-device AR creation and discovery scaffold

Spatial Spray

Location-aware AR spray creation workflow for iPhone and Apple Vision Pro.

Spatial Spray is a cross-device AR spray creation and discovery product scaffold. It separates social sign-in, unique usernames, nearby discovery, AR brush behavior, UGC moderation, and Apple native build requests into contracts, API services, web simulation, native source, and Mac Builder workflow. The current goal is to validate product flow and engineering boundaries, not to claim production social-network or App Store readiness.

Identity before publishing

Users establish identity and claim a unique username before publishing.

Apple, Google, and Facebook login are modeled through dev-mode adapters; production provider token verification remains server-side work.

Location-indexed AR pieces

Spray pieces are defined by location, surface pose, and AR anchor metadata.

The API and contracts record geo point, surface pose, spray strokes, visibility, and moderation state.

Moderated UGC workflow

Reporting, blocking, deletion, and location denial are modeled from the first scaffold.

AR UGC is not only a brush effect; it also needs discovery, public-space safety, and review workflows.

Problem Architecture Workflow

Problem

AR social creation is not only a rendering problem.

When users create AR content around real indoor, outdoor, or public-building contexts, the product must answer who published the content, where it is discoverable, how nearby users find it, how inappropriate content is reported, whether certain places should deny creation, and how iPhone and Vision Pro clients share the same data model.

Native iOS and visionOS execution still requires macOS, Xcode, ARKit, RealityKit, signing assets, and device testing. This repository validates product rules through Linux-runnable API, contracts, and web simulation before handing native build requests to the Apple toolchain through Mac Builder workflow.

Product risk it reduces

Before investing in devices, location precision, App Store review, and production login providers, the product loop can validate identity, nearby discovery, spray behavior, and moderation workflow.

What it is

An engineering scaffold for a cross-device AR UGC product.

Spatial Spray splits the core product loop into API, brush engine, web simulator, native Apple scaffold, and Mac Builder handoff. Product rules stay testable on Linux, while native AR behavior remains Apple-platform work.

Moderation workflow

UGC governance entrypoint

Report, block, owner delete, admin hide/remove, audit log, and location denylist are part of the scaffold.

What it is not

The repository is a product and engineering scaffold, not a production promise.

These boundaries prevent the web simulator, dev login, and native scaffold from being mistaken for launch-ready capability.

Not a production social-network backend; the current API uses local persistent state.
Not an App Store ready app; signing, privacy materials, device testing, and review preparation remain required.
Not a real provider-login integration; production must verify Apple, Google, and Facebook tokens server-side.
Not a global centimeter-accurate AR anchoring system; MVP anchors use manual-local data and surface pose modeling.
Not an ARKit replacement inside the browser; the web simulator validates product and API flow.
Does not encourage real-world graffiti or property damage; spray pieces are digital AR content.

Operator workflow

Local product loop and Mac Builder validation

The local flow validates contracts, API behavior, web simulation, and end-to-end tests. Native build requests first target a mock builder; real iOS and visionOS builds still require a remote Mac.

  1. Install dependencies: `pnpm install`.
  2. Run workflow checks: `pnpm workflow:check`.
  3. Run tests: `pnpm test`.
  4. Start local API and web simulator: `pnpm dev`.
  5. Start mock builder: `pnpm native:mac-builder:mock`.
  6. Submit a visionOS build request: `SPATIAL_SPRAY_MAC_BUILDER_URL=http://127.0.0.1:4391 pnpm native:visionos-build:submit`.
